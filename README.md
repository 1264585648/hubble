# Hubble

> 一个面向企业告警场景的通用告警机器人框架：接入告警、理解告警、调用工具、推送结论，并在群聊中继续追问和处置。

Hubble 的目标不是再做一个简单的 Webhook 转发器，而是做一个可插拔的 **AI Alert Bot Runtime**：

- 支持 Webhook、轮询任务、自定义数据源等多种告警接入方式。
- 引入大模型对告警进行摘要、分级、归因、降噪和处置建议生成。
- 提供工具调用层，可接入日志、数据库、监控接口、CI/CD、知识库等内部系统。
- 支持飞书、企业微信、钉钉、Slack、通用 Webhook 等推送通道。
- 支持会话监听，在可回复的群聊环境中继续追问、补充上下文和执行工具。

## 最小运行方式

```bash
pip install -e .
uvicorn hubble.server:app --reload
```

发送测试告警：

```bash
curl -X POST http://127.0.0.1:8000/webhook/demo \
  -H 'Content-Type: application/json' \
  -d @examples/webhook_payload.json
```

## 第一阶段目标

1. 完成统一告警事件模型 `AlertEvent`。
2. 完成 Webhook 接入和轮询任务接入。
3. 完成模型调度层接口，先提供 Echo / OpenAI-compatible 两种 Provider。
4. 完成工具注册表和安全执行器。
5. 完成飞书、企业微信、通用 Webhook 推送。
6. 完成飞书/CLI 会话监听，让用户可以在群里追问：
   - “为什么会这样？”
   - “帮我查最近 10 分钟日志。”
   - “影响范围是什么？”
   - “给我生成处理步骤。”

## 设计原则

- **插件优先**：所有接入、模型、工具、推送、会话都通过统一接口注册。
- **结构化优先**：模型输出必须尽可能结构化，方便后续推送、审计和自动化动作。
- **安全优先**：工具调用默认只读，写操作必须显式授权、审计和二次确认。
- **渐进增强**：没有大模型时也能作为普通告警路由器运行；引入大模型后增强分析能力。
- **云原生友好**：后续支持 Docker、Kubernetes、Helm、Prometheus Metrics。
